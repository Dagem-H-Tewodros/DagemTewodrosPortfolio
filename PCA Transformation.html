Data Engineering: Taming High-Cardinality Categories with PCA

How dimensionality reduction transformed my insurance model from a computational nightmare into an elegant, performant system

Introduction

After validating provincial differences through rigorous A/B testing, I was ready to build predictive models. There was just one problem: the curse of dimensionality.

Looking at my feature set, the numbers were intimidating:

  • ZipCode: 200+ unique values
  • Make (vehicle manufacturer): 52 unique values
  • Province: 9 unique values
  • CoverCategory: 18 unique values
  • Plus 15+ other categorical variables

One-hot encoding all these variables would create over 500 binary features. My laptop would struggle. Training times would be measured in hours. Model files would be massive. Overfitting would be inevitable.

I needed a solution that could:

  1. Reduce dimensionality dramatically
  2. Retain the information content
  3. Work in production (fast inference)
  4. Maintain interpretability where possible

The answer was Principal Component Analysis (PCA) but not the way it's typically taught.

In this post, I'll show you how I applied column-specific PCA transformation to reduce 500+ features to 55 principal components while retaining 95% of variance. If you've ever struggled with high-cardinality categorical variables, this is for you.

Why PCA? Understanding The technique

What is Principal Component Analysis?

PCA transforms correlated features into uncorrelated principal components that capture maximum variance.

In simple terms:

  • Take 50 correlated binary features (from one-hot encoding)
  • Find the "directions" in feature space with most variation
  • Keep only the top directions that explain most variance
  • Result: 10-15 components instead of 50 features

The Math

For a data matrix "X" with n samples and p features:

  1. Center the data: Subtract column means
  2. Compute covariance matrix: C = (1/n) X^T X
  3. Find eigenvalues and eigenvectors of C
  4. Select top k eigenvectors (principal components)
  5. Transform data: X_reduced = X · PC_k

The magic: k components can explain 90-95% of variance from p features when k << p.

Why Column-Specific PCA?

I made a critical architectural decision: apply PCA separately to each high-cardinality column rather than globally.

Advantages:

  • Preserves semantic meaning (ZipCode components separate from Make components)
  • Easier to interpret and debug
  • More control over compression ratio per column
  • Better for production (update one column's PCA without retraining everything)

Trade-off:

  • Potentially misses global correlations
  • More PCA models to manage

For my use case, the interpretability and modularity benefits outweighed the trade-offs.

Implementaion: Step-by-Step PCA Pipeline

Step 1: Environment Setup

import numpy as np
import joblib
import os

from sklearn.decomposition import PCA
import matplotlib.pyplot as plt
import seaborn as sns

# Set random seed for reproducibility
np.random.seed(42)

# Create directory for saved models
os.makedirs('../saved_models/pca', exist_ok=True)

Step 2: Define the PCA Pipeline Function


def apply_pca_to_column(df, column_name, explained_variance=0.95, 
                        save_path=None):
    """
    Apply PCA transformation to a single high-cardinality column.
    
    Args:
        df: DataFrame containing the data
        column_name: Name of column to transform
        explained_variance: Target cumulative explained variance (0-1)
        save_path: Path to save the PCA models
        
    Returns:
        transformed_array: PCA-transformed features
        models_dict: Dictionary containing encoder, scaler, and PCA
    """
    
    print(f"\n{'='*60}")
    print(f"Processing column: {column_name}")
    print(f"{'='*60}")
    
    # Step 1: One-Hot Encode
    print(f"Step 1: One-hot encoding...")
    encoder = OneHotEncoder(
        sparse_output=False,
        handle_unknown='ignore',  # Critical for production
        drop=None  # Keep all categories
    )
    
    encoded = encoder.fit_transform(df[[column_name]])
    n_categories = encoded.shape[1]
    print(f"  Original categories: {df[column_name].nunique()}")
    print(f"  Encoded features: {n_categories}")
    
    # Step 2: Scale the encoded features
    print(f"Step 2: Scaling encoded features...")
    scaler = RobustScaler()  # Robust to outliers
    scaled = scaler.fit_transform(encoded)
    
    # Step 3: Apply PCA
    print(f"Step 3: Applying PCA...")
    pca = PCA(n_components=explained_variance, random_state=42)
    transformed = pca.fit_transform(scaled)
    
    n_components = pca.n_components_
    variance_explained = pca.explained_variance_ratio_.sum()
    
    print(f"  Components selected: {n_components}")
    print(f"  Variance explained: {variance_explained:.1%}")
    print(f"  Compression ratio: {n_categories}/{n_components} = "
          f"{n_categories/n_components:.1f}x")
    
    # Step 4: Package models for production
    models_dict = {
        'encoder': encoder,
        'scaler': scaler,
        'pca': pca,
        'n_components': n_components,
        'explained_variance': pca.explained_variance_ratio_,
        'cumulative_variance': np.cumsum(pca.explained_variance_ratio_),
        'original_features': n_categories,
        'column_name': column_name
    }
    
    # Step 5: Save models
    if save_path:
        model_path = os.path.join(save_path, f'{column_name}_pca.joblib')
        joblib.dump(models_dict, model_path)
        print(f"  Models saved to: {model_path}")
    
    return transformed, models_dict

Step 3: Apply to High-Cardinality Columns


# Define columns for PCA transformation
pca_columns = ['ZipCode', 'make', 'Province', 'CoverCategory']

# Storage for transformed data and models
pca_models = {}
transformed_dfs = []

# Apply PCA to each column
for col in pca_columns:
    transformed, models = apply_pca_to_column(
        df, 
        col, 
        explained_variance=0.95,
        save_path='../saved_models/pca'
    )
    
    # Store models
    pca_models[col] = models
    
    # Create DataFrame with component columns
    component_cols = [f'{col}_PC{i+1}' for i in range(models['n_components'])]
    transformed_df = pd.DataFrame(
        transformed,
        columns=component_cols,
        index=df.index
    )
    
    transformed_dfs.append(transformed_df)

# Combine all transformed features
df_pca = pd.concat(transformed_dfs, axis=1)
print(f"\nTotal PCA features created: {df_pca.shape[1]}")

Production Deployment: Transformating New Data

The Critical Challenge

In production, new customer data arrives with raw categorical values. We need to apply the *exact same* PCA transformation used during training.

Loading Saved PCA Models


# Load all PCA models
pca_models_loaded = {}
for col in pca_columns:
    model_path = f'../saved_models/pca/{col}_pca.joblib'
    pca_models_loaded[col] = joblib.load(model_path)
    print(f"✅ Loaded PCA models for: {col}")

Transformation Function for New Data


def transform_new_customer(customer_data, pca_models):
    """
    Apply PCA transformation to new customer data.
    
    Args:
        customer_data: Dictionary with customer information
        pca_models: Dictionary of saved PCA models
        
    Returns:
        DataFrame with transformed features
    """
    
    # Convert to DataFrame
    df_new = pd.DataFrame([customer_data])
    
    # Initialize result DataFrame
    df_transformed = pd.DataFrame(index=[0])
    
    # Transform each PCA column
    for col, models in pca_models.items():
        if col in df_new.columns:
            # Step 1: Encode
            encoded = models['encoder'].transform(df_new[[col]])
            
            # Step 2: Scale
            scaled = models['scaler'].transform(encoded)
            
            # Step 3: PCA
            transformed = models['pca'].transform(scaled)
            
            # Step 4: Add to result
            for i in range(models['n_components']):
                df_transformed[f'{col}_PC{i+1}'] = transformed[0, i]
        else:
            print(f"⚠️  Warning: Column {col} not in customer data")
    
    return df_transformed

Advanced: Incrmental PCA for Large Datasets

For datasets too large to fit in memory:


from sklearn.decomposition import IncrementalPCA

def incremental_pca_transform(df, column_name, batch_size=10000,
                              explained_variance=0.95):
    """
    Apply PCA incrementally for large datasets.
    """
    
    # Encode and scale first (must fit in memory)
    encoder = OneHotEncoder(sparse_output=False, handle_unknown='ignore')
    scaler = RobustScaler()
    
    encoded = encoder.fit_transform(df[[column_name]])
    scaled = scaler.fit_transform(encoded)
    
    # Incremental PCA
    n_components = min(int(encoded.shape[1] * explained_variance), 
                       encoded.shape[1])
    
    ipca = IncrementalPCA(n_components=n_components, batch_size=batch_size)
    
    # Fit in batches
    n_batches = int(np.ceil(scaled.shape[0] / batch_size))
    for i in range(n_batches):
        start_idx = i * batch_size
        end_idx = min((i + 1) * batch_size, scaled.shape[0])
        ipca.partial_fit(scaled[start_idx:end_idx])
    
    # Transform all data
    transformed = ipca.transform(scaled)
    
    return transformed, {
        'encoder': encoder,
        'scaler': scaler,
        'pca': ipca
    }

Common Pitfalls and Solutions

Pitfall 1: forgetting to Scale

  • Problem: PCA is sensitive to scale. Unscaled features dominate.
  • Solution: Always scale after one-hot encoding.

# ❌ WRONG
pca = PCA(n_components=0.95)
transformed = pca.fit_transform(encoded)  # No scaling!

# ✅ CORRECT
scaler = RobustScaler()
scaled = scaler.fit_transform(encoded)
pca = PCA(n_components=0.95)
transformed = pca.fit_transform(scaled)

Pitfall 2: Applying PCA to Target Variable

  • Problem: Including target in PCA causes data leakage.
  • Solution: Only transform features, never targets.

# ❌ WRONG
X_with_target = df[features + ['HasClaim']]
pca.fit_transform(X_with_target)

# ✅ CORRECT
X_only = df[features]
pca.fit_transform(X_only)

Pitfall 3: Not Saving Tarnsformers fro Production

  • Problem: Can't transform new data without saved models.
  • Solution: Save encoder, scaler, AND pca using joblib.

# ✅ CORRECT
models = {
    'encoder': encoder,
    'scaler': scaler,
    'pca': pca
}
joblib.dump(models, 'column_pca_models.joblib')

Pitfalls 4: Using Wrong Number of Components

  • Problem: Too few components lose information; too many defeat the purpose.
  • Solution: Use scree plots and variance explained thresholds (95% is common).

Alternative Approaches Considered

Why Not Other Dimensionality Reduction Techniques?

  1. t-SNE
    • Non-deterministic (different each run)
    • Computationally expensive
    • Can't transform new data easily
    • Good for visualization only
  2. UMAP
    • Requires additional library
    • Hyperparameter tuning needed
    • Less interpretable than PCA
    • Better for non-linear relationships
  3. Feature Selection (Chi-square, Mutual Information)
    • Discards information completely
    • Doesn't handle redundancy well
    • Maintains interpretability
    • Fast and simple
  4. Entity Embeddigs
    • Requires neural network training
    • More complex to implement
    • Less transparent
    • Can capture complex patterns
  • PCA was Chosen Because
    • Deterministic and reproducible
    • Fast to compute
    • Easy to deploy in production
    • Well-understood and documented
    • Built into scikit-learn
    • 95% variance retention balances compression and information

    • Comparable or better accuracy

When to use Column-Specific PCA

  • Use when
    • High-cardinality categorical variables (50+ categories)
    • Computational constraints (memory/time)
    • Production deployment requirements
    • Semantic separation desired (province separate from vehicle make)
  • Don't use when
    • Low-dimensional data already (less than 20 features)
    • Feature interpretability is critical
    • Non-linear relationships dominate
    • Tree-based models (they handle high dimensions well)
